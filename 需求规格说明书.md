## 1. 项目背景与目标

### 1.1 背景
华南师范大学各学院（含研究院、研究中心）的讲座、学术报告信息分散发布在本单位二级站点（如 `psy.scnu.edu.cn`、`cs.scnu.edu.cn`）的「通知公告」「学术活动预告」等栏目中，校级学术活动则由社会科学处（`skc.scnu.edu.cn`）等职能部门集中发布。师生若想全面了解近期校内讲座，必须逐个访问学院站点翻看，效率低、容易遗漏。

### 1.2 目标
建设一个面向华师师生的**讲座信息聚合网站**：
- 自动/半自动地从各学院、研究院与校级来源采集讲座信息；
- 以**时间线**形式统一呈现；
- 支持**按学院（含校区）筛选**；
- 支持**关键词模糊搜索**（标题、主题、主讲人等）。

### 1.3 范围边界（已确认全量）
- **纳入**：华师全部教学学院、学部、**研究院、研究中心**，以及校级学术活动栏目中，明确为「讲座 / 学术报告 / 学术沙龙 / 讲坛」类、且有具体时间地点的活动。范围以 `scnu.edu.cn/sitemap` 站点导航为准（含四校区及校级科研机构）。
- **暂不纳入（初版）**：会议通知、招生/招聘/答辩公告、培训报名等非讲座类通知；不替代官方发布系统；不做在线报名/签到。

## 3. 数据来源与现状分析（基于调研）

### 3.1 学院与校区（来自 `scnu.edu.cn/sitemap`）
华师教学科研机构分四个校区，SRS 中以「校区 → 学院」两级组织：

- **石牌校园**：教育科学学院、哲学与社会发展学院、马克思主义学院、历史文化学院、外国语言文化学院、国际文化学院、美术学院、教育信息技术学院、数学科学学院、生命科学学院、地理科学学院、计算机学院、心理学院等
- **大学城校园**：文学院、经济与管理学院、法学院、政治与公共管理学院、体育科学学院、音乐学院、物理学院、化学学院、旅游管理学院、信息光电子科技学院等
- **佛山校区（南海校园）**：国际商学院、软件学院、工学部、阿伯丁数据科学与人工智能学院等
- **汕尾校区（滨海校园）**：基础教育学院、数据科学与工程学院、材料与新能源学院、创意设计学院等
- **校级科研机构 / 研究院 / 研究中心**：站点导航中大量「研究中心 / 研究院」已确认一并纳入聚合范围。

### 3.2 信息发布规律（实测样例）
- **列表页 URL 规律**：各学院站点多为同一套国产 CMS，文章路径形如 `http://psy.scnu.edu.cn/a/20260421/2947.html`（`/a/YYYYMMDD/ID.html`），列表分页为 `/2.html`、`/3.html`。
- **讲座混在通知中**：「通知公告」栏目同时含招生、答辩、公示等，讲座条目靠标题关键词（如「学术讲座」「讲座」「学术沙龙」）与正文时间字段识别。
- **详情页字段（实测心理学院样例）**：
  - 标题（常含「X月X日 主讲人 教授学术讲座」）
  - 发布时间（如 `2026-04-21 14:42:00`）、来源（学院名）
  - 讲座时间（中文，如「4月25日（星期六）上午10:00-11:00」）
  - 地点（如「心理学院213会议室」）
  - 题目（讲座主题，如「审美客体心理表征的认知加工机制」）
  - 主讲人及单位/职称（如「沈模卫 教授（浙江大学心理与行为科学系）」）
  - 主讲人简介
  - 主办单位（通常为栏目来源）
- **校级来源**：社会科学处「学术活动预告」（`skc.scnu.edu.cn/xueshuhuodongyugao/`）是天然的聚合源，但实测抓取存在网络/证书问题，需单独处理（UA、超时、重试、必要时降级）。

### 3.3 关键结论
不同学院 CMS 模板相似但**并非完全一致**，讲座时间常以中文非结构化文本出现——采集层必须为每类来源维护**可配置的解析规则**，而非写死单一模板。

### 3.4 视觉样式参考（AI Hot，已确认）
前端整体复刻 AI Hot（aihot.virxact.com）的「极客阅读器」风格，具体要求：
- **配色**：浅色为主（背景 `#ffffff` / `#f7f8fa`），主文字 `#1a1a1a`，次要文字（时间/来源）`#6b7280`；强调色用橙红 `#ff6600`（用于「即将开始」高亮、热度），链接/主操作色用蓝 `#2563eb`。
- **布局**：单栏居中，最大宽约 `760px`；顶部为站点标题 + 横向文字链导航（全部 / 按校区）+ 右侧圆角搜索框。
- **信息流**：按**日期分组**的垂直时间线（如「7月8日 周二」作分隔），组内按讲座时间倒序；条块间细线分隔。
- **单条卡片字段排布**：左「讲座时间 + 学院(来源) + 校区/状态标签」，标题（粗，16–18px），摘要（主讲人 · 地点 · 题目，14px），底部胶囊标签（学院 / 校区），以及「主讲人简介 / 原文链接」灰色小字。
- **标签**：浅灰底 `#f3f4f6`、深字 `#374151`、圆角 4–6px 的胶囊（pill）。
- **字体**：系统无衬线栈，中文 `PingFang SC / Microsoft YaHei`，基号 14–16px，行高约 1.6。
- **响应式**：移动端单列、导航与筛选可横向滚动。

## 4. 功能需求

### FR1 讲座数据采集（采集器）
- **FR1.1** 从可配置的来源列表（学院 + 研究院 + 校级栏目）抓取列表页与详情页。
- **FR1.2** 识别讲座条目：结合标题关键词 + 正文是否含时间地点字段，过滤非讲座通知。
- **FR1.3** 解析为标准字段（见第 6 节数据模型），并对中文时间文本做结构化（转为 `YYYY-MM-DD HH:MM`）。
- **FR1.4** 去重：以源站 URL 为唯一键，重复抓取自动更新而非新增。
- **FR1.5** 增量更新：记录最后抓取位点，避免全量重抓；支持失败重试与断点续抓。
- **FR1.6** 采集规则可配置：每来源可指定列表页 URL 模板、分页规律、详情页字段选择器。
- **FR1.7** 每日自动定时抓取（本地通过系统计划任务 / 脚本调度，初版先手动可跑 + 提供调度说明）。

### FR2 时间线展示
- **FR2.1** 默认按日期分组、组内按讲座时间倒序呈现时间线。
- **FR2.2** 每条以卡片展示：讲座时间、标题、学院/校区、地点、主讲人、题目。
- **FR2.3** 状态区分：已结束 / 进行中 / 即将开始（7 天内橙红高亮）。

### FR3 按学院筛选
- **FR3.1** 顶部导航/标签按校区（全部 / 石牌 / 大学城 / 佛山 / 汕尾 / 校级）切换；再按学院筛选。
- **FR3.2** 多选学院进行「或」筛选（初版可先单选 + 校区过滤）。

### FR4 模糊搜索
- **FR4.1** 关键词搜索覆盖：标题、题目、主讲人、主讲人单位、简介。
- **FR4.2** 支持主讲人姓名、主题词、单位名的模糊匹配（前端包含匹配即可，分词/拼音为增强项）。

### FR5 讲座详情页
- **FR5.1** 展示完整字段（含主讲人简介、原文链接）。
- **FR5.2** 标注数据来源与采集时间，链接回官方原文。

### FR6 数据管理后台（初版可简化 / 二期）
- **FR6.1** 手动新增 / 编辑 / 删除讲座（弥补爬虫漏抓或解析错误）。
- **FR6.2** 标记某条为「无效 / 已取消」。

## 5. 非功能需求

- **NF1 性能**：列表首屏加载 < 2s；前端分页或虚拟滚动。
- **NF2 可用性**：响应式布局，移动端友好（师生主要场景）。
- **NF3 准确性**：讲座识别准确率目标 ≥ 90%（人工抽检评估）；时间结构化准确率优先。
- **NF4 可维护性**：采集规则与来源列表配置化，新增来源无需改代码。
- **NF5 更新频率**：每日定时增量抓取（本地调度）。
- **NF6 合规性**：仅聚合公开信息；每条保留官方原文链接与来源署名；不做商业利用。
- **NF7 健壮性**：单源抓取失败不影响其他源；错误有日志与告警。

## 6. 数据模型（草案）

```
Lecture {
  id            // 内部主键
  sourceUrl     // 源站文章 URL（唯一键，用于去重）
  college       // 学院/来源名称
  campus        // 校区：石牌/大学城/佛山/汕尾/校级
  title         // 列表标题
  topic         // 讲座题目
  lectureStart  // 结构化开始时间 datetime
  lectureEnd    // 结构化结束时间（可选）
  location      // 地点
  speaker       // 主讲人
  speakerAffiliation // 主讲人单位/职称
  speakerBio    // 主讲人简介
  organizer     // 主办单位
  publishTime   // 官方发布时间
  crawledAt     // 采集时间
  status        // 正常/已取消/待审核
}
```
采集器输出为 `data/lectures.json`（数组），前端直接加载。

## 7. 技术路线（已确认：方案 A）

**架构**：Python 采集器（requests + BeautifulSoup）定时抓取 → 生成 `data/lectures.json` → 纯静态前端（原生 HTML/CSS/JS）读取 JSON 渲染时间线/筛选/搜索。本地用 `python -m http.server` 起服务做演示；日后可整体平移至云/校内静态托管。

**每日自动**：本地通过操作系统计划任务（Windows 任务计划程序 / cron）每日调用采集器；脚本自身支持手动运行与增量更新。

**为何方案 A**：契合「聚合 + 展示」核心诉求，零后端、部署极简、易维护；二期若要订阅提醒/用户系统再升级方案 B（带后端数据库）。

## 9. 关键风险

1. **解析差异**：各学院 CMS 模板细节不同，需逐站适配解析规则，工作量随来源数线性增长。
2. **时间结构化**：中文「4月25日 上午10:00」需可靠转 datetime，模糊/缺字段需兜底（年份默认当年）。
3. **讲座识别准确率**：通知与讲座混排，关键词误判可能漏抓或误抓。
4. **访问稳定性**：部分站点（如社科处）存在证书/网络问题；需处理超时、重试、UA。
5. **反爬**：高频抓取可能被限，需限速与 UA 策略。

## 10. 验收标准（草案）

- 试点来源讲座采集准确率 ≥ 90%，时间字段解析正确率 ≥ 95%；
- 时间线按日期分组、组内按时间正确排序，状态标识正确；
- 校区/学院筛选结果与源站一致；
- 关键词搜索能命中标题/题目/主讲人；
- 详情页可跳转官方原文；
- 本地 `http` 服务可正常打开并交互。
